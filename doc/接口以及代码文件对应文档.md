# 接口以及代码文件对应文档

## 一、通用约定

### 1）统一前缀

```Bash
/api/v1
```

### 2）统一返回结构

建议所有接口返回：

```JSON
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

- `code = 0`：成功

- `code \!= 0`：失败

- `message`：提示信息

- `data`：业务数据

### 3）分页结构

```JSON
{
  "items": [],
  "total": 100,
  "page": 1,
  "page_size": 20
}
```

### 4）认证方式

- `Authorization: Bearer \&lt;access\_token\&gt;`

- `tenant\_id` 不由前端直接传，默认从登录态中解析

## 二、认证接口 `auth`

### 1\. 登录

#### 接口

```HTTP
POST /api/v1/auth/login
```

#### 请求字段

|字段|类型|必填|说明|
|---|---|---|---|
|email|string|是|登录邮箱|
|password|string|是|登录密码|

#### 请求示例

```JSON
{
  "email": "admin@example.com",
  "password": "123456"
}
```

#### 响应字段

|字段|类型|说明|
|---|---|---|
|access\_token|string|访问令牌|
|refresh\_token|string|刷新令牌|
|token\_type|string|固定为 bearer|
|user|object|当前用户信息|

#### `user` 结构

|字段|类型|说明|
|---|---|---|
|id|string|用户ID|
|tenant\_id|string|租户ID|
|username|string|用户名|
|email|string|邮箱|
|role|string|角色|

#### 响应示例

```JSON
{
  "code": 0,
  "message": "success",
  "data": {
    "access_token": "xxx",
    "refresh_token": "yyy",
    "token_type": "bearer",
    "user": {
      "id": "u1",
      "tenant_id": "t1",
      "username": "admin",
      "email": "admin@example.com",
      "role": "tenant_admin"
    }
  }
}
```

### 2\. 刷新 token

#### 接口

```HTTP
POST /api/v1/auth/refresh
```

#### 请求字段

|字段|类型|必填|说明|
|---|---|---|---|
|refresh\_token|string|是|刷新令牌|

#### 响应字段

|字段|类型|说明|
|---|---|---|
|access\_token|string|新访问令牌|
|refresh\_token|string|新刷新令牌|
|token\_type|string|bearer|

### 3\. 获取当前用户信息

#### 接口

```HTTP
GET /api/v1/auth/me
```

#### 响应字段

|字段|类型|说明|
|---|---|---|
|id|string|用户ID|
|tenant\_id|string|租户ID|
|username|string|用户名|
|email|string|邮箱|
|role|string|角色|
|status|string|状态|

### 4\. 退出登录

#### 接口

```HTTP
POST /api/v1/auth/logout
```

#### 请求字段

通常不需要额外字段，服务端使 token 失效即可。

## 十二、最推荐你优先实现的接口顺序

### 第一批必须先做

1. `POST /auth/login`

2. `GET /auth/me`

3. `POST /knowledge\-bases`

4. `POST /documents/upload`

5. `POST /conversations/\{conversation\_id\}/messages`

6. `POST /tickets`

7. `GET /audit\-logs`

### 第二批补齐

8. `GET /documents`

9. `GET /tickets`

10. `PATCH /tickets/\{ticket\_id\}/status`

11. `GET /metrics/dashboard`

### 第三批增强

12. `agent/run`

13. `agent/graph`

14. `audit\-logs/summary`

15. `knowledge\-bases/\{kb\_id\}/stats`

## 二、后端目录：每个文件职责说明表

### 1）`backend/app/main\.py`

**职责：**

- FastAPI 应用入口

- 注册中间件

- 挂载路由

- 启动时初始化资源

- 关闭时释放资源

**典型内容：**

- `app = FastAPI\(\.\.\.\)`

- CORS 配置

- 路由聚合 `include\_router`

- startup / shutdown 事件

### 2）`backend/app/core/config\.py`

**职责：**

- 统一管理配置项

- 读取环境变量

- 区分开发/测试/生产环境

**典型内容：**

- 数据库 URL

- Redis URL

- Qdrant URL

- JWT 密钥

- LLM API Key

- 环境名称

### 3）`backend/app/core/logging\.py`

**职责：**

- 初始化日志系统

- 统一日志格式

- 支持结构化日志输出

**典型内容：**

- logger 配置

- request\_id 注入

- error log / info log / audit log

### 4）`backend/app/core/exceptions\.py`

**职责：**

- 定义项目统一异常

- 统一错误响应格式

- 封装业务异常和权限异常

**典型内容：**

- `BusinessException`

- `UnauthorizedException`

- `ForbiddenException`

- 全局异常处理器

### 5）`backend/app/core/security\.py`

**职责：**

- JWT 生成与解析

- 密码哈希与验证

- 登录认证相关安全逻辑

**典型内容：**

- `create\_access\_token`

- `create\_refresh\_token`

- `verify\_password`

- `hash\_password`

### 6）`backend/app/core/context\.py`

**职责：**

- 保存请求级上下文

- 管理当前 `tenant\_id`

- 管理当前 `user\_id`

- 方便服务层、仓储层自动读取上下文

**典型内容：**

- `ContextVar`

- `current\_tenant\_id`

- `current\_user\_id`

- request 结束后清理上下文

### 7）`backend/app/db/base\.py`

**职责：**

- 定义 ORM 基类

- 提供所有模型的公共父类

**典型内容：**

- `Base = DeclarativeBase`

- 公共 mixin
