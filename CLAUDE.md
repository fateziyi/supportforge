# SupportForge — 企业级 SaaS 智能客服 AI 平台开发规范

> 所有代码编写、功能开发、逻辑优化必须严格遵守本文档规则，请先通读全文再编写代码。

## 一、项目整体定位

| 维度 | 说明 |
|------|------|
| **项目用途** | 面向企业 SaaS 软件的智能客服助手系统 |
| **核心价值** | 实战项目、秋招面试核心项目、全栈 AI Agent 商业闭环项目 |
| **项目性质** | **多租户 SaaS 平台**，非简单聊天机器人、非纯 Prompt 演示项目 |
| **开发原则** | 优先完成业务闭环，前端够用即可，重心放在后端、RAG 知识库、AI 智能工作流 |

## 三、项目核心业务流程

```
用户提交客服问题
  → 系统自动分类问题类型
  → 检索租户专属知识库内容
  → AI 生成标准回复建议
  → 复杂问题自动流转人工工单
  → 全流程日志审计、行为溯源
  → 租户数据完全隔离
```

## 六、多租户强制开发规则（最高优先级）

1. **所有业务表必须携带 `tenant_id` 租户标识**
2. 后端所有查询、新增、修改、删除 **强制过滤租户 ID**
3. 租户身份仅从登录 JWT 令牌解析，**绝不信任前端传参**
4. 禁止普通租户查询其他租户任何数据
5. 平台超级管理员拥有全局查看权限，操作必须留审计日志
6. 知识库、文档、对话、工单、日志全部归属独立租户

---

## 七、RBAC 权限角色规范

| 角色 | 英文标识 | 权限范围 |
|------|----------|----------|
| 平台超级管理员 | `platform_admin` | 管理所有租户、全局配置 |
| 租户管理员 | `tenant_admin` | 管理本租户用户、知识库、工单 |
| 客服坐席 | `support_agent` | 使用对话、处理工单、查询知识库 |
| 审计人员 | `auditor` | 仅只读查看日志与流程记录 |

### 接口权限校验顺序

1. 校验登录态（JWT）
2. 校验租户权限（tenant_id）
3. 校验角色权限（role）
4. 无权限统一返回标准错误结构

---

## 八、后端代码编写强制规范

1. **路由层**：只做参数接收与响应返回，禁止写业务逻辑
2. **Service 层**：业务逻辑统一放入 service 服务层
3. **Repository 层**：数据库操作统一放入 repository 仓储层
4. 请求响应严格使用 **Pydantic 模型**约束
5. 统一全局异常捕获，禁止直接抛出原生错误
6. 接口返回格式全局统一（前后端约定一致）
7. 大量 IO、文档解析、向量入库全部走 **异步任务**
8. 优先使用异步语法编写 FastAPI 接口

---

## 九、数据库开发规范

1. 所有表必须包含 `created_at`、`updated_at` 时间字段
2. 租户相关表建立联合索引：`tenant_id + 常用筛选字段`
3. 所有表结构修改必须使用 **Alembic 迁移脚本**，禁止手动改库
4. 逻辑删除优先，物理删除谨慎使用
5. 向量库 Qdrant 存储切片文本 + 租户 ID 元数据，检索强制过滤租户

---

## 十、RAG 知识库开发规范

### 标准流程

```
文档上传 → 文本解析 → 文本切片 → 向量化 → 存入向量库
  → 语义检索 → 重排择优 → 拼接上下文 → AI 生成回答
```

### 核心规则

1. 回答优先保证准确，严控 AI 幻觉
2. 所有回答必须附带知识库来源引用
3. 检索置信度过低直接转入人工工单，不强行编造答案
4. 切片规则统一，前后逻辑一致

---

## 十一、LangGraph AI 智能体开发规范

1. 使用节点式拆分工作流，每个节点功能单一
2. 全局状态统一管理，流转清晰易懂
3. 支持流程中断、人工介入、流程恢复
4. 智能体无权跨租户调取任何数据
5. 问题判定复杂、信息不足、业务敏感 → 直接触发工单流转

### 固定智能体节点

| 节点 | 职责 |
|------|------|
| 问题分类节点 | 判断问题类型（FAQ / 复杂 / 紧急） |
| 意图识别节点 | 解析用户真实意图 |
| 知识库检索节点 | RAG 语义检索租户专属知识库 |
| 智能回复生成节点 | 基于检索结果生成标准回复 |
| 工单创建节点 | 低置信度 / 复杂问题转人工工单 |
| 审计日志记录节点 | 记录全流程操作与决策 |

---

## 十二、工单流转规则

### 自动转人工工单条件

1. 问题难度超出知识库覆盖范围
2. AI 置信度低于设定阈值
3. 需要后台系统操作处理
4. 用户明确要求人工客服
5. 敏感业务、高风险咨询内容

### 工单必须留存信息

- 对话上下文
- 检索依据
- 问题摘要
- 流转原因

## 十三、全链路审计日志规范

### 必须记录字段

- `tenant_id`（租户 ID）
- `user_id`（用户 ID）
- `action`（操作行为）
- `resource_type` + `resource_id`（操作资源）
- `created_at`（时间戳）
- 请求摘要、AI 调用信息、耗时、Token 消耗、错误信息

### 必记录行为

- 登录登出
- 文档上传
- 知识库修改
- AI 对话
- 工单操作
- 权限访问
- 越权访问尝试

## 十四、前端开发统一规范

1. 全程 TS 严格类型，禁止 `any` 泛滥
2. 页面布局统一风格，复用全局布局组件
3. 所有接口请求统一封装，统一处理 loading、报错、超时
4. 登录态、权限、租户信息全局统一管理
5. 页面只做展示与交互，复杂计算与业务判断全部交给后端
6. 严格使用 shadcn/ui + Tailwind，少写自定义冗余样式

### 前端必备页面

- 登录页
- 数据大盘（Dashboard）
- 租户管理
- 知识库管理
- 文档管理
- 客服对话面板
- 工单管理
- 审计日志
- 个人设置

## 十六、AI 编码行为约束

1. 新增功能先判断归属模块，不随意新建文件
2. 新增数据表优先判断是否需要租户隔离
3. 改动接口同步更新入参、出参、前端调用处
4. 尽量增量开发，不擅自重构整体架构
5. 命名跟随项目现有风格，保持全局统一
6. 不确定业务逻辑优先提问，不自行脑补开发
7. 编写代码附带详细注释，每个函数必须有职责说明、关键参数说明、设计意图说明，方便初学者理解和面试口述
8. 优先编写符合面试讲解逻辑的代码结构，方便秋招口述项目
9. 注释使用中文，代码关键字和变量名使用英文

## 十八、Git 分支与开发流程

> 个人项目，简化分支策略，直接在 main 分支开发。

### 分支模型

```
main        ← 直接开发，每次 push 触发 CI 检查
feature/*   ← 可选：复杂功能开发时拉分支，完成后合回 main
```

- 日常开发直接 push 到 `main`
- 复杂功能或实验性改动可拉 `feature/*` 分支，完成后合回 main
- commit message 格式：`[模块名] 功能描述`，如 `[auth] 添加 JWT 登录接口`

## 十九、CI/CD 流水线

> GitHub Actions 自动化，push 即触发，保证代码质量。

### 流水线设计

| Workflow | 触发条件 | 内容 |
|----------|----------|------|
| `ci-backend.yml` | push 涉及 `backend/` | Ruff Lint + Black + MyPy + pytest |
| `ci-frontend.yml` | push 涉及 `frontend/` | ESLint + TypeScript + Build Check |
| `build.yml` | push to main | Docker 构建验证（不推送） |
| `deploy.yml` | push to main（可选） | Docker 构建推送 + SSH 部署 |

### GitHub Secrets（部署时配置）

| Secret | 说明 |
|--------|------|
| `DOCKER_USERNAME` | Docker Hub 用户名 |
| `DOCKER_PASSWORD` | Docker Hub 密码 |
| `DEPLOY_HOST` | 部署服务器 IP |
| `DEPLOY_USER` | SSH 用户名 |
| `DEPLOY_KEY` | SSH 私钥 |
| `DEPLOY_PATH` | 项目部署路径 |
