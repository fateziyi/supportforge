## 一、项目整体定位

| 维度 | 说明 |
|------|------|
| **项目用途** | 面向企业 SaaS 软件的智能客服助手系统 |
| **核心价值** | 实战项目、秋招面试核心项目、全栈 AI Agent 商业闭环项目 |
| **项目性质** | **多租户 SaaS 平台**，非简单聊天机器人、非纯 Prompt 演示项目 |

## 三、项目核心业务流程

```
用户提交客服问题
  → 系统自动分类问题类型
  → 检索租户专属知识库内容
  → AI 生成标准回复建议

  → 全流程日志审计、行为溯源
  → 租户数据完全隔离
```

## 六、多租户强制开发规则（最高优先级）

1. **所有业务表必须携带 `tenant_id` 租户标识**
2. 后端所有查询、新增、修改、删除 **强制过滤租户 ID**
3. 租户身份仅从登录 JWT 令牌解析，**绝不信任前端传参**
4. 禁止普通租户查询其他租户任何数据
5. 平台超级管理员拥有全局查看权限，操作必须留审计日志
6. 知识库、文档、对话、工单、日志全部归属独立租户
7. **关联归属一致性**：文档→知识库、工单→对话/处理人、消息→对话、审计→用户、Agent Run→对话/消息等关联资源必须属于同一租户；优先使用 `(id, tenant_id)` 组合外键，Service 层仍须校验。
8. `tenant_id` 查询条件必须由 Repository/Service 统一注入；禁止只按资源 `id` 查询后直接返回或修改。
9. 平台配置、Alembic 元数据等非业务表可不带 `tenant_id`，但必须在模型或迁移中说明例外理由。

## 七、RBAC 权限角色规范

| 角色 | 英文标识 | 权限范围 |
|------|----------|----------|
| 平台超级管理员 | `platform_admin` | 管理所有租户、全局配置 |
| 租户管理员 | `tenant_admin` | 管理本租户用户、知识库、工单 |
| 客服坐席 | `support_agent` | 使用对话、处理工单、查询知识库 |
| 审计人员 | `auditor` | 仅只读查看日志与流程记录 |

### 接口权限校验顺序

1. 校验登录态（JWT）
2. 校验租户权限（tenant_id）
3. 校验角色权限（role）
4. 无权限统一返回标准错误结构

### RBAC 权限矩阵要求

- 新增业务接口前必须在接口文档中标注允许角色、读写操作和数据范围（本租户 / 平台全局）。
- `platform_admin` 的跨租户操作必须显式分支处理，记录目标租户与操作原因；不得通过放宽普通租户查询条件实现。
- `auditor` 仅可读取审计、流程与统计数据，不得调用任何写接口或触发 Agent 写操作。

## 八、后端代码编写强制规范

1. **路由层**：只做参数接收与响应返回，禁止写业务逻辑
2. **Service 层**：业务逻辑统一放入 service 服务层
3. **Repository 层**：数据库操作统一放入 repository 仓储层
4. 请求响应严格使用 **Pydantic 模型**约束
5. 统一全局异常捕获，禁止直接抛出原生错误
6. 接口返回格式全局统一（前后端约定一致）
7. 大量 IO、文档解析、向量入库全部走 **异步任务**
8. 优先使用异步语法编写 FastAPI 接口

### API 契约与认证安全

1. 业务接口成功响应统一为 `{"code": 0, "message": "success", "data": ...}`；失败响应统一为 `{"code": <业务码>, "message": <安全提示>, "data": null}`。
2. 接口文档必须定义请求字段校验、分页/排序、时间格式、错误码和所需角色；流式接口须明确 SSE 事件类型与断线处理方式。
3. `request_id` 必须可在响应头、日志、审计记录和 Agent Trace 中关联。
4. 密码只允许使用 Argon2 等专用慢哈希；禁止使用明文、MD5、SHA-1、SHA-256 作为正式密码存储方案。
5. 生产环境必须配置非默认 JWT 密钥；mock 登录、测试密钥和演示账号只能在 development/test 使用。
6. JWT 必须包含 `sub`、`tenant_id`、`role`、`token_type`、`iat`、`exp`；不得包含密码、密码哈希、密钥或未经处理的用户输入。

## 九、数据库开发规范

1. 所有表必须包含 `created_at`、`updated_at` 时间字段
2. 租户相关表建立联合索引：`tenant_id + 常用筛选字段`
3. 所有表结构修改必须使用 **Alembic 迁移脚本**，禁止手动改库
4. 逻辑删除优先，物理删除谨慎使用
5. 向量库 Qdrant 存储切片文本 + 租户 ID 元数据，检索强制过滤租户
6. 已在共享环境执行的迁移禁止修改、删除或重写；结构变更必须新增 Alembic 迁移并执行 `alembic upgrade head` 验证。
7. 每次新增外键、唯一约束或索引，都要同时更新 ORM metadata、迁移脚本与数据表设计文档，避免自动迁移出现漂移。

## 十、RAG 知识库开发规范

### 标准流程

```
文档上传 → 文本解析 → 文本切片 → 向量化 → 存入向量库
  → 语义检索 → 重排择优 → 拼接上下文 → AI 生成回答
```

### 核心规则

1. 回答优先保证准确，严控 AI 幻觉
2. 所有回答必须附带知识库来源引用
3. 检索置信度过低直接转入人工工单，不强行编造答案
4. 切片规则统一，前后逻辑一致
5. 文档重解析、向量 upsert、删除向量必须可幂等，并以 `tenant_id`、`knowledge_base_id`、`document_id` 作为 Qdrant payload 过滤边界。

## 十一、LangGraph AI 智能体开发规范

1. 使用节点式拆分工作流，每个节点功能单一
2. 全局状态统一管理，流转清晰易懂
3. 支持流程中断、人工介入、流程恢复
4. 智能体无权跨租户调取任何数据
5. 问题判定复杂、信息不足、业务敏感 → 直接触发工单流转
6. 用户输入、检索文档和模型输出均视为不可信数据；检索文本不得覆盖系统规则或直接驱动高风险操作。
7. Agent 工具必须由服务端白名单注册，参数使用 Pydantic 校验；禁止模型拼接 SQL、文件路径或任意 HTTP 请求。

### 固定智能体节点

| 节点 | 职责 |
|------|------|
| 问题分类节点 | 判断问题类型（FAQ / 复杂 / 紧急） |
| 意图识别节点 | 解析用户真实意图 |
| 知识库检索节点 | RAG 语义检索租户专属知识库 |
| 智能回复生成节点 | 基于检索结果生成标准回复 |
| 工单创建节点 | 低置信度 / 复杂问题转人工工单 |
| 审计日志记录节点 | 记录全流程操作与决策 |

## 十二、工单流转规则

### 自动转人工工单条件

1. 问题难度超出知识库覆盖范围
2. AI 置信度低于设定阈值
3. 需要后台系统操作处理
4. 用户明确要求人工客服
5. 敏感业务、高风险咨询内容

### 工单必须留存信息

- 对话上下文
- 检索依据
- 问题摘要
- 流转原因

## 十三、全链路审计日志规范

### 必须记录字段

- `tenant_id`（租户 ID）
- `user_id`（用户 ID）
- `action`（操作行为）
- `resource_type` + `resource_id`（操作资源）
- `created_at`（时间戳）
- 请求摘要、AI 调用信息、耗时、Token 消耗、错误信息

### 审计与隐私补充规则

- 日志、审计和 Trace 中不得记录密码、JWT、API Key 等秘密；手机号、邮箱等个人信息按最小必要原则脱敏或仅保存摘要。
- 审计日志默认只追加不修改；若需清理或归档，必须记录执行人、范围、原因和时间。
- 需要保存 Prompt、检索文本或模型输出时，先判断是否含敏感信息，并在接口/模型层定义保留期限。

### 必记录行为

- 登录登出
- 文档上传
- 知识库修改
- AI 对话
- 工单操作
- 权限访问
- 越权访问尝试

## 十四、前端开发统一规范

1. 全程 TS 严格类型，禁止 `any` 泛滥
2. 页面布局统一风格，复用全局布局组件
3. 所有接口请求统一封装，统一处理 loading、报错、超时
4. 登录态、权限、租户信息全局统一管理
5. 页面只做展示与交互，复杂计算与业务判断全部交给后端
6. 严格使用 shadcn/ui + Tailwind，少写自定义冗余样式
7. 前端请求工具必须统一注入 Access Token、解析 `{code, message, data}`、保留 `X-Request-ID`，并统一处理 401、403、超时和业务错误。

### 前端必备页面

- 登录页
- 数据大盘（Dashboard）
- 租户管理
- 知识库管理
- 文档管理
- 客服对话面板

## 十六、AI 编码行为约束

1. 新增功能先判断归属模块，不随意新建文件
2. 新增数据表优先判断是否需要租户隔离
3. 改动接口同步更新入参、出参、前端调用处
4. 尽量增量开发，不擅自重构整体架构
5. 命名跟随项目现有风格，保持全局统一
6. 不确定业务逻辑优先提问，不自行脑补开发
7. 编写代码附带详细注释，每个函数必须有职责说明、关键参数说明、设计意图说明，方便初学者理解和面试口述
8. 优先编写符合面试讲解逻辑的代码结构，方便秋招口述项目
9. 注释使用中文，代码关键字和变量名使用英文
10. **Spec 完成后必须回写实现记录**：当某个 Day 的 spec 中所有需求开发完毕并验证通过后，必须在 spec 文件末尾追加「实现完成记录」章节，记录实际产出、验证结果、与 spec 的偏差，供后续开发参考和回溯
11. 开发每个 Day 前先确认当前 Alembic head、已有 API 契约、对应 Spec 与 `PROGRESS.md`；禁止基于过期 mock 行为扩展正式业务逻辑。
12. 新增或修改业务功能必须同步补充自动化测试；最低质量门禁为 `ruff`、`black`、`mypy`、`pytest`，CI 不得因缺少测试目录而静默跳过。
13. Celery 等异步任务必须定义幂等键、状态机、最大重试次数、退避策略、失败原因和人工重试入口，并写入审计日志。

## 十七、Spec 实现完成记录规范

当某个 Day 的 spec 全部需求开发完毕并验证通过后，必须在对应 spec 文件末尾追加如下结构的「实现完成记录」章节：

```markdown
## N. 实现完成记录

> 本章节记录 Day X spec 实际完成情况，包含产出文件、验证结果、与 spec 的偏差说明。

### 完成时间

YYYY-MM-DD

### 实际产出文件

| 文件 | 说明 |
|------|------|
| `backend/app/xxx.py` | 一句话描述该文件的核心职责和关键字段/功能 |

### 验证结果

- ✅ 具体验证命令 + 预期结果（如 `ruff check app/` → All checks passed!）
- ✅ 功能验证（如 curl 登录返回 200 + mock token）
- ✅ 数据库验证（如 `\dt` 显示 N 张表）

### 与 Spec 的偏差

1. **偏差描述**：如果实作与 spec 定义有不同，说明原因和影响
2. 如果无偏差，写"与 Spec 无偏差"
```

### 必记内容

- **实际产出文件**：每个新增/修改的文件都要列出来，附一句话说明
- **验证结果**：lint、启动、功能、数据库等关键验证点都要列出来
- **偏差说明**：如果实作与 spec 有差异（如字段名改变、导入方式改变、额外依赖新增），必须如实记录原因

### 不记内容

- 不记录开发过程中的中间尝试（只记最终结果）
- 不记录 lint auto-fix 的琐碎细节（只记是否有手动修复的重要偏差）

## 十八、Git 分支与开发流程

> 个人项目，简化分支策略，直接在 main 分支开发。

### 分支模型

```
main        ← 直接开发，每次 push 触发 CI 检查
feature/*   ← 可选：复杂功能开发时拉分支，完成后合回 main
```

- 日常开发直接 push 到 `main`
- 复杂功能或实验性改动可拉 `feature/*` 分支，完成后合回 main
- commit message 格式：`[模块名] 功能描述`，如 `[auth] 添加 JWT 登录接口`

## 十九、CI/CD 流水线

> GitHub Actions 自动化，push 即触发，保证代码质量。

### 流水线设计

| Workflow | 触发条件 | 内容 |
|----------|----------|------|
| `ci-backend.yml` | push 涉及 `backend/` | Ruff Lint + Black + MyPy + pytest |

| `build.yml` | push to main | 容器构建验证（不推送） |
| `deploy.yml` | push to main（可选） | 容器构建推送 + SSH 部署 |

### GitHub Secrets（部署时配置）

| Secret | 说明 |
|--------|------|
| `DOCKER_USERNAME` | 容器镜像仓库用户名 |
| `DOCKER_PASSWORD` | 容器镜像仓库密码 |
| `DEPLOY_HOST` | 部署服务器 IP |
| `DEPLOY_USER` | SSH 用户名 |
| `DEPLOY_KEY` | SSH 私钥 |
| `DEPLOY_PATH` | 项目部署路径 |
